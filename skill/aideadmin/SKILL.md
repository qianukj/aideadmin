# AideAdmin Framework

## Core Rule

Build for AI-readable business systems. Keep implementation predictable: fixed directories, unified API contracts, server-driven permissions, React-only frontends, installer-first database setup, and black/white minimalist admin UX.

Never introduce non-React frontend stacks, legacy PHP admin references, or traditional left-tree menu UX.

Brand asset: use `assets/aideadmin.png` when generated docs, websites, or README content need the official AideAdmin logo.

## Quick Workflow

1. Identify the target surface: `server`, `admin`, `h5`, installer, docs, generator, or Docker.
2. Read only the reference needed for the task:
   - Architecture and module layout: `references/architecture.md`
   - Backend/API/RBAC/database/security rules: `references/backend-rules.md`
   - React PC/H5 UI rules and style inheritance: `references/frontend-rules.md`
   - Customer IM and third-party SDK integrations: `references/integrations.md`
   - Local run, build, Docker, and release: `references/delivery.md`
3. Follow the standard development path:
   `table -> model -> repository -> service -> controller -> route -> nav-entry -> api -> page`.
4. Keep generated code explicit and maintainable. Avoid hidden metaprogramming and page-local theme systems.
5. Validate security and delivery rules before finishing.

## Non-negotiables

- Backend: Go + Gin + GORM + MySQL + Redis.
- Frontend: React + TypeScript + Vite for both PC admin and H5.
- API response: `{ code, show, msg, data }`.
- Admin UX: top menu + navigation icon cards, not a traditional permission-tree menu.
- Customer IM: global floating dock and message monitoring, not an independent menu item.
- Database install: first run enters installer; do not require database connection until installer submits config.
- Table prefix: installer submits `database.table_prefix`; migrations/seeds use `{{prefix}}`.
- License: MIT, free commercial use allowed.

## Business Module Pattern

When implementing a new admin module:

1. Create logical table names such as `cms_article`; physical names are generated by `{table_prefix}cms_article`.
2. Add Go files under `server/internal/admin/{dto,vo,repository,service,controller,route}` and models under `server/internal/model`.
3. Use DTO validation in controllers; put business rules and transactions in services; put data access in repositories.
4. Register routes under `/adminapi/<module>/<resource>/<action>`.
5. Create React API files and pages:
   - `admin/src/api/<module>/<resource>.ts`
   - `admin/src/pages/<module>/<resource>/index.tsx`
   - H5 files only when that business flow needs mobile access.
6. Add top navigation, icon-card entry, button permissions, and API permission strings.
7. Add tests or API verification proportional to the change risk.

## Security Baseline

Always include these capabilities in framework-level code:

- Redis-backed rate limiting by IP, user, route, and scene.
- Request debounce and duplicate-submit locks in frontend request wrappers.
- `Idempotency-Key` / nonce validation for high-risk writes.
- SQL injection prevention through parameter binding and field allowlists.
- Upload validation by size, extension, MIME, and generated object key.
- DDoS baseline through body limits, trusted proxy handling, degradation switches, and deployment behind CDN/WAF/LB for production.

## Output Style

For code generation, provide:

- File paths and concise implementation notes.
- Permission strings and route paths.
- DTO/list/export contracts when relevant.
- Database logical table names and prefix-safe migration notes.
- React parent component/style inheritance notes.

For documentation generation, keep docs bilingual-ready, AI-readable, and consistent with the framework terminology.
